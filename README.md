# 🟦 Day 1 – What is Go?

Go (Golang) is a modern programming language created at Google.  
Its development began in **2007**, and it was open-sourced in **2009**.

Go was created with one clear objective:

**Make programming simple, fast, and reliable without unnecessary complexity.**

It blends:
- the **performance & safety** of compiled languages like C/C++,  
- with the **ease & flexibility** of dynamic languages like Python.

Because of this, Go is clean, minimal, and excellent for building fast, scalable applications—especially on multi-core systems.

---

## ⭐ Why Should You Learn Go?

### 1) Beginner-Friendly  
Go is extremely easy to learn.  
Its syntax is minimal and beginner-friendly, and the tools are powerful yet simple.

### 2) Fast & Efficient  
Go compiles down to machine code.  
This makes it extremely fast—perfect for distributed systems like:

- Kubernetes  
- Docker  
- Cloudflare  

### 3) Simple but Powerful  
Go has **only 25 keywords**, yet provides powerful features like:
- interfaces  
- fast concurrency  
- strong typing  

### 4) Amazing Career Opportunities  
Go is rapidly growing across startups, cloud platforms, DevOps tools, and enterprises.  
Go developers are highly in demand with strong salaries.

### Popular use cases:
- Backend systems  
- Microservices  
- DevOps tooling  
- Cloud engineering  
- High-performance systems  

---

## ⭐ Why Go Was Created  

Go was created to fix limitations in older languages:

1. **Slow compilation** (like C++)  
2. **Overly complex codebases**  
3. **Weak concurrency models**

Go solves these problems by offering:
- very fast compilation  
- clean & readable syntax  
- built-in concurrency (goroutines)  
- single-binary deployment  

---

## 🟨 Key Features of Go

- **Simple syntax** (only 25 keywords)  
- **C-level performance**  
- **Memory efficient**  
- **Built-in concurrency (goroutines)**  
- **Easy deployment (single binary)**  
- **Cross-platform builds**

## 🟩 Where is Go Used?

Some of the world’s most powerful tools are written in Go:

Go is the top choice for:

# 🟨 Day 3: Strings & Formatting

Strings are one of the most used data types in Go, but they behave a bit differently compared to languages like C++ or Java — so it's worth understanding them properly early on.

## 🔤 What is a String in Go?

A string in Go is an **immutable sequence of bytes** — once created, it cannot be modified in place. Any "change" actually creates a new string.

```
name := "Golang"
// name[0] = 'X' // ❌ Error: cannot assign to name[0]
```

### Double Quotes vs Backticks

**Interpreted string (escape characters work):**
```
normal := "Line 1\nLine 2"
```

**Raw string (printed exactly as written):**
```
raw := `Line 1
Line 2`
```

Backtick strings are especially useful for multi-line templates, regex patterns, or SQL queries — no need to escape anything.

## 🔡 Rune vs Byte

- **byte** → alias for `uint8`, represents a single ASCII character (0–255)
- **rune** → alias for `int32`, represents a single Unicode code point (supports emojis, accented characters, non-Latin scripts, etc.)

```
word := "café"

fmt.Println(len(word))          // byte length, not character count
fmt.Println(len([]rune(word))) // actual character count
```

This distinction becomes critical when working with non-ASCII text — relying only on `len()` can give you the wrong result.

## 🧵 Iterating Over a String

```
for i, ch := range "Go❤️" {
    fmt.Printf("Index: %d, Char: %c\n", i, ch)
}
```

`range` automatically decodes UTF-8 and gives you each rune correctly, instead of raw bytes.

## 🖨️ Formatting Verbs — The Core of the `fmt` Package

| Verb | Purpose |
|------|---------|
| `%v` | default format of the value |
| `%+v` | struct value with field names |
| `%T` | type of the value |
| `%d` | integer |
| `%f` | float (6 decimal places by default) |
| `%.2f` | float with 2 decimal precision |
| `%s` | string |
| `%q` | double-quoted string |
| `%t` | boolean |
| `%x` | hexadecimal |

```
type User struct {
    Name string
    Age  int
}

u := User{"Manish", 28}

fmt.Printf("%v\n", u)  // {Manish 28}
fmt.Printf("%+v\n", u) // {Name:Manish Age:28}
fmt.Printf("%T\n", u)  // main.User
```

## 📦 The `Print` Family

| Function | Returns a value? | Adds newline? |
|----------|-------------------|----------------|
| `Print` | No (writes to console) | No |
| `Println` | No | Yes |
| `Printf` | No | You add `\n` manually |
| `Sprint` | Yes (returns a string) | No |
| `Sprintf` | Yes | You add `\n` manually |
| `Fprintf` | Writes to a writer (file, buffer) | You add `\n` manually |

```
msg := fmt.Sprintf("User %s logged in after %.1f seconds", "Manish", 2.347)
```

`Sprintf` is the most commonly used one in real projects — perfect for building formatted strings, error messages, or API responses.

## 🧩 Common Functions in the `strings` Package

```
import "strings"

strings.ToUpper("rishikesh")            // RISHIKESH
strings.Contains("Aatm Yoga", "Yoga")   // true
strings.Split("a,b,c", ",")             // [a b c]
strings.TrimSpace("  hello  ")          // "hello"
strings.ReplaceAll("foo-bar", "-", "_") // foo_bar
strings.Join([]string{"a", "b"}, "-")   // a-b
```

These functions come up constantly in everyday coding — cleaning user input, generating slugs, formatting text, and more.

# 🟪 Day 4: Control Flow

Go's control flow is simple, but a few things work differently compared to other languages — for example, parentheses are never required.

## 🔀 If / Else

```
score := 82

if score >= 90 {
    fmt.Println("Grade A")
} else if score >= 75 {
    fmt.Println("Grade B")
} else {
    fmt.Println("Grade C")
}
```

### If with an Init Statement

Declaring a variable and checking a condition in the same line — a very common pattern in Go code.

```
if err := doSomething(); err != nil {
    fmt.Println("Error:", err)
}
```

Here `err` is only valid inside the `if` block — the scope stays tight and clean.

## 🔁 Switch

Go's switch is smart — once a case matches, it automatically breaks. There's no accidental fallthrough like in C.

```
day := 3

switch day {
case 1, 7:
    fmt.Println("Weekend")
case 2, 3, 4, 5, 6:
    fmt.Println("Weekday")
default:
    fmt.Println("Invalid day")
}
```

### Condition-less Switch (a clean replacement for if-else chains)

```
age := 25

switch {
case age < 13:
    fmt.Println("Child")
case age < 20:
    fmt.Println("Teenager")
default:
    fmt.Println("Adult")
}
```

### The `fallthrough` Keyword

Use this when you deliberately want the next case to run as well:

```
switch grade := "B"; grade {
case "A":
    fmt.Println("Excellent")
    fallthrough
case "B":
    fmt.Println("Good job")
}
```

## 🔄 For Loop — Go's Only Loop

Go doesn't have separate `while` or `do-while` loops — everything is built using `for`.

```
// Classic for loop
for i := 0; i < 5; i++ {
    fmt.Println(i)
}
```

```
// while-style loop
count := 0
for count < 5 {
    count++
}
```

```
// infinite loop
for {
    fmt.Println("This will keep running until break")
    break
}
```

### Looping with Range (slices, maps, strings)

```
fruits := []string{"Apple", "Mango", "Banana"}

for index, value := range fruits {
    fmt.Println(index, value)
}
```

## ⏸️ Break and Continue

```
for i := 1; i <= 10; i++ {
    if i%2 == 0 {
        continue // skip even numbers
    }
    if i > 7 {
        break // stop the loop after 7
    }
    fmt.Println(i)
}
```

### Labeled Break — for Nested Loops

```
outer:
for i := 0; i < 3; i++ {
    for j := 0; j < 3; j++ {
        if j == 1 {
            break outer
        }
        fmt.Println(i, j)
    }
}
```

## ⏳ Defer — The Most Misunderstood Keyword

`defer` postpones the execution of a statement until the surrounding function returns.

```
func readFile() {
    fmt.Println("File opened")
    defer fmt.Println("File closed") // this runs last
    fmt.Println("Reading file contents")
}
```

**Output:**
```
File opened
Reading file contents
File closed
```

### Multiple Defers Behave Like a Stack (LIFO)

```
defer fmt.Println("1")
defer fmt.Println("2")
defer fmt.Println("3")
// Output order: 3, 2, 1
```

In real-world code, `defer` is almost always used for resource cleanup — closing files, closing database connections, unlocking mutexes — keeping the code leak-proof.

# 🟥 Day 5: Functions

Functions are the backbone of Go — they're treated as first-class citizens, meaning you can store them in variables and pass them around like any other value.

## 🧱 Basic Function Declaration

```
func greet(name string) string {
    return "Hello, " + name
}
```

## 🎯 Multiple Return Values

One of Go's signature features — a function can return more than one value. This is the foundation of Go's error handling pattern.

```
func divide(a, b int) (int, error) {
    if b == 0 {
        return 0, fmt.Errorf("cannot divide by zero")
    }
    return a / b, nil
}

result, err := divide(10, 2)
if err != nil {
    fmt.Println("Error:", err)
} else {
    fmt.Println("Result:", result)
}
```

## 🏷️ Named Return Values

```
func stats(nums []int) (sum, avg int) {
    for _, n := range nums {
        sum += n
    }
    avg = sum / len(nums)
    return // naked return — both named values are returned automatically
}
```

⚠️ Named returns are handy for short functions, but they can hurt readability in longer ones — use them carefully.

## 🌀 Variadic Functions

Useful when you don't know how many arguments will be passed:

```
func total(nums ...int) int {
    sum := 0
    for _, n := range nums {
        sum += n
    }
    return sum
}

total(1, 2, 3)        // 6
total(10, 20, 30, 40) // 100
```

`fmt.Println` itself is a variadic function — that's why you can pass as many arguments as you want to it.

## 🎭 Anonymous Functions

A function with no name, used immediately:

```
square := func(x int) int {
    return x * x
}

fmt.Println(square(5)) // 25
```

You can also invoke it inline:

```
func() {
    fmt.Println("This runs immediately")
}()
```

## 🔒 Closures — Functions That Remember State

A closure is a function that "captures" variables from outside its own body.

```
func counter() func() int {
    count := 0
    return func() int {
        count++
        return count
    }
}

next := counter()
fmt.Println(next()) // 1
fmt.Println(next()) // 2
fmt.Println(next()) // 3
```

Here, `count` stays alive even after the outer function has returned — that's the real power of closures. Rate limiters, counters, and middleware logic are all built on this pattern.

## 🧩 Higher-Order Functions

Functions that take other functions as arguments, or return them:

```
func applyDiscount(price float64, discountFn func(float64) float64) float64 {
    return discountFn(price)
}

festiveOffer := func(p float64) float64 {
    return p * 0.8 // 20% off
}

finalPrice := applyDiscount(1000, festiveOffer)
fmt.Println(finalPrice) // 800
```

This pattern is great whenever you need flexible, plug-and-play logic — pricing strategies, validators, or callbacks.

# 🟫 Day 6: Packages, Modules & Workspaces

Understanding Go's project structure is essential for keeping code organized and reusable.

## 📦 What is a Package?

Every `.go` file belongs to a package. Files in the same folder usually belong to the same package.

```
package main // an executable program
```

```
package utils // a reusable library package
```

Only the `main` package can be run directly with `go run` — every other package is imported and used elsewhere.

## 📁 Modules and go.mod

A module is a collection of packages with versioning. To start a new project:

```
go mod init github.com/manishkandari9/yoga-scraper
```

This creates a `go.mod` file:

```
module github.com/manishkandari9/yoga-scraper

go 1.21

require (
    github.com/some/package v1.2.0
)
```

## 📥 Installing Dependencies

```
go get github.com/joho/godotenv
```

To clean up unused dependencies:

```
go mod tidy
```

## 🔓 Exported vs Unexported

Go decides "public" or "private" purely based on naming convention:

```
package pricing

func CalculateTax(amount float64) float64 { // Capitalized = Exported (public)
    return amount * 0.18
}

func roundOff(val float64) float64 { // lowercase = unexported (private)
    return val
}
```

`CalculateTax` can be accessed from other packages, while `roundOff` stays private to the `pricing` package.

## 🗂️ Creating Your Own Package

```
myproject/
├── go.mod
├── main.go
└── pricing/
    └── tax.go
```

```
// pricing/tax.go
package pricing

func CalculateTax(amount float64) float64 {
    return amount * 0.18
}
```

```
// main.go
package main

import (
    "fmt"
    "myproject/pricing"
)

func main() {
    fmt.Println(pricing.CalculateTax(1000))
}
```

## 🧵 Multi-Module Workspaces (Go 1.18+)

When you're working across multiple modules at once and need to link them locally without a `replace` directive:

```
go work init
```

```
go work use ./service-a
go work use ./service-b
```

This creates a `go.work` file that treats all listed modules as a single workspace — very handy for larger, microservice-style projects.

# 🟦 Day 7: Core Data Structures

## 📏 Arrays — Fixed-Size Collections

```
var scores [5]int
scores = [5]int{90, 85, 78, 92, 88}

fmt.Println(len(scores)) // 5
```

An array's size is part of its type — `[5]int` and `[3]int` are completely different types and can't be assigned to each other.

## 📐 Slices — Go's Dynamic Array

Slices are built on top of arrays and can grow dynamically.

```
nums := []int{1, 2, 3}
nums = append(nums, 4, 5)

fmt.Println(nums)      // [1 2 3 4 5]
fmt.Println(len(nums)) // length
fmt.Println(cap(nums)) // capacity
```

### Creating a Slice with make()

```
buffer := make([]int, 0, 10) // length 0, capacity 10
```

When you already know the expected size, pre-allocating with `make` is a performance win — it avoids repeated reallocation.

### Slicing Syntax

```
data := []int{10, 20, 30, 40, 50}

fmt.Println(data[1:3]) // [20 30]
fmt.Println(data[:2])  // [10 20]
fmt.Println(data[2:])  // [30 40 50]
```

⚠️ **Important gotcha:** Slices are reference types — if you create one slice from another (without using `copy()`), both share the same underlying array.

```
original := []int{1, 2, 3}
alias := original[:2]
alias[0] = 99

fmt.Println(original) // [99 2 3] — original changed too!
```

If you need a fully independent copy:

```
safe := make([]int, len(original))
copy(safe, original)
```

## 🗺️ Maps — Key-Value Pairs

```
inventory := map[string]int{
    "yoga-mat":           50,
    "meditation-cushion": 20,
}

inventory["blocks"] = 30 // add
delete(inventory, "blocks") // remove

value, exists := inventory["yoga-mat"]
if exists {
    fmt.Println("Stock:", value)
}
```

Maps are unordered — iteration order is never guaranteed and can differ on every run.

## ⚖️ Deep vs Shallow Copy

- **Slices and Maps → Reference types** (shallow copy by default)
- **Arrays and Structs → Value types** (assignment creates a full deep copy)

```
type Point struct{ X, Y int }

p1 := Point{1, 2}
p2 := p1     // deep copy
p2.X = 100
fmt.Println(p1.X) // 1 — p1 stays unaffected
```

```
s1 := []int{1, 2}
s2 := s1     // shallow copy (same underlying array)
s2[0] = 100
fmt.Println(s1[0]) // 100 — s1 is affected too!
```

Understanding this difference is crucial for avoiding bugs — this exact mistake is one of the most common in production Go code.

# 🟩 Day 8: Structs & Methods

## 🧱 Struct — Your Custom Data Type

```
type Client struct {
    Name    string
    Website string
    Active  bool
}

c := Client{
    Name:    "Aatm Yogashala",
    Website: "aatmyogashala.com",
    Active:  true,
}
```

## 🧬 Embedding — Go's "Inheritance-Like" Feature

Go doesn't have classical inheritance — instead, it favors **composition/embedding**.

```
type BaseInfo struct {
    CreatedAt string
    UpdatedBy string
}

type BlogPost struct {
    BaseInfo // embedded struct
    Title    string
    Content  string
}

post := BlogPost{
    BaseInfo: BaseInfo{CreatedAt: "2026-07-01", UpdatedBy: "Manish"},
    Title:    "7-Day Yoga Retreat Guide",
}

fmt.Println(post.CreatedAt) // accessed directly, as if it were its own field
```

## ⚙️ Methods — Attaching Functions to a Struct

```
func (c Client) Summary() string {
    return fmt.Sprintf("%s (%s)", c.Name, c.Website)
}

fmt.Println(c.Summary())
```

### Value Receiver vs Pointer Receiver — The Most Important Concept Here

```
// Value receiver — operates on a copy, original stays unchanged
func (c Client) Deactivate() {
    c.Active = false
}
```

```
// Pointer receiver — modifies the original struct
func (c *Client) DeactivatePermanently() {
    c.Active = false
}
```

```
c.Deactivate()
fmt.Println(c.Active) // true (no effect on the original)

c.DeactivatePermanently()
fmt.Println(c.Active) // false (permanently changed)
```

**Rule of thumb:** If a method needs to modify the struct's state, or the struct is large (making copies expensive), always use a pointer receiver.

## 🔖 JSON Marshalling — Essential for APIs/Configs

```
type Course struct {
    Title    string `json:"title"`
    Duration int    `json:"duration_days"`
}

course := Course{Title: "200hr TTC", Duration: 30}

jsonData, _ := json.Marshal(course)
fmt.Println(string(jsonData))
// {"title":"200hr TTC","duration_days":30}
```

```
var parsed Course
json.Unmarshal(jsonData, &parsed)
```

Struct tags (`json:"title"`) define what the JSON field will be called — without them, Go uses the field name exactly as written.

# 🟨 Day 9: Pointers

## 📍 What is a Pointer?

A pointer is a variable that stores the **memory address** of another variable.

```
age := 28
ptr := &age // & = address-of operator

fmt.Println(ptr)  // memory address, e.g. 0xc0000140a0
fmt.Println(*ptr) // 28 — * = dereference operator (get the value)
```

## ✏️ Modifying a Value Through a Pointer

```
func increaseAge(p *int) {
    *p = *p + 1
}

age := 28
increaseAge(&age)
fmt.Println(age) // 29
```

Without passing a pointer, `age` would just be copied into the function, and the original `age` would never change.

## ❌ Pointer Arithmetic is Not Allowed

Unlike C/C++, you can't do `ptr + 1` in Go — this is intentionally disallowed to keep the language safe.

## 🎯 When Should You Use a Pointer?

| Situation | What to Use |
|-----------|--------------|
| Small value, read-only | Value |
| Large struct, or needs modification | Pointer |
| A method needs to change the original data | Pointer receiver |
| You need the flexibility of nil checks | Pointer |

```
type Config struct {
    Timeout int
    Debug   bool
}

func loadConfig() *Config {
    return &Config{Timeout: 30, Debug: false}
}
```

In real projects — when a function returns a struct that will later be modified, or is large, returning a pointer is the better practice.

# 🟪 Day 10: Interfaces

Interfaces handle Go's polymorphism — without needing classes.

## 🧩 What Does an Interface Define?

An interface defines only **behavior (methods)**, never implementation.

```
type Notifier interface {
    Send(message string) error
}
```

Any type that has a `Send(message string) error` method automatically satisfies the `Notifier` interface — no `implements` keyword required (implicit implementation).

```
type EmailService struct{}
func (e EmailService) Send(message string) error {
    fmt.Println("Email sent:", message)
    return nil
}

type WhatsAppService struct{}
func (w WhatsAppService) Send(message string) error {
    fmt.Println("WhatsApp sent:", message)
    return nil
}

func notifyUser(n Notifier, msg string) {
    n.Send(msg)
}

notifyUser(EmailService{}, "Booking confirmed!")
notifyUser(WhatsAppService{}, "Booking confirmed!")
```

This is the core benefit of decoupling — `notifyUser` doesn't care how the notification is actually sent. Adding a new channel (SMS, Telegram) is as simple as creating a new type that implements `Send`.

## 🌀 The Empty Interface (`any`)

```
func printAnything(val any) {
    fmt.Println(val)
}

printAnything(42)
printAnything("Rishikesh")
printAnything(true)
```

`any` (the Go 1.18+ alias for `interface{}`) can hold a value of any type — useful when the type isn't known in advance.

## 🔍 Type Assertion

```
var data any = "Hello Go"

str, ok := data.(string)
if ok {
    fmt.Println("It's a string:", str)
}
```

## 🔀 Type Switch

```
func describe(val any) {
    switch v := val.(type) {
    case int:
        fmt.Println("It's an integer:", v)
    case string:
        fmt.Println("It's a string:", v)
    default:
        fmt.Println("Unknown type:", v)
    }
}
```

## 💉 Dependency Injection Pattern

```
type Database interface {
    GetUser(id int) string
}

type Service struct {
    db Database // depends on the interface, not a concrete type
}

func NewService(db Database) *Service {
    return &Service{db: db}
}
```

This makes testing much easier — pass a real database in production, and a mock/fake database in tests, without changing anything in `Service`.

# 🟥 Day 11: Error Handling

Go doesn't have try/catch — errors are handled explicitly, as regular values.

## ⚠️ Basic Error Pattern

```
func fetchPage(url string) (string, error) {
    if url == "" {
        return "", errors.New("url cannot be empty")
    }
    return "page content", nil
}

content, err := fetchPage("")
if err != nil {
    fmt.Println("An error occurred:", err)
    return
}
fmt.Println(content)
```

## 📝 Adding Context with fmt.Errorf

```
func loadSitemap(path string) error {
    if _, err := os.Open(path); err != nil {
        return fmt.Errorf("failed to load sitemap %s: %w", path, err)
    }
    return nil
}
```

`%w` **wraps** the error — the original error information is preserved, and can later be checked using `errors.Is` / `errors.Unwrap`.

## 🎯 Sentinel Errors

```
var ErrNotFound = errors.New("record not found")

func findClient(id int) error {
    return ErrNotFound
}

if errors.Is(findClient(5), ErrNotFound) {
    fmt.Println("This client doesn't exist")
}
```

## 🧩 Custom Error Types

```
type ValidationError struct {
    Field string
    Msg   string
}

func (v ValidationError) Error() string {
    return fmt.Sprintf("%s: %s", v.Field, v.Msg)
}

func validateForm(email string) error {
    if !strings.Contains(email, "@") {
        return ValidationError{Field: "email", Msg: "invalid format"}
    }
    return nil
}

err := validateForm("wrongemail.com")
var valErr ValidationError
if errors.As(err, &valErr) {
    fmt.Println("Field:", valErr.Field)
}
```

## 💥 Panic and Recover

Only use `panic` when the program genuinely cannot continue (an unrecoverable situation).

```
func safeDivide(a, b int) (result int) {
    defer func() {
        if r := recover(); r != nil {
            fmt.Println("Recovered from panic:", r)
            result = 0
        }
    }()

    return a / b // if b == 0, this panics, and the defer above recovers it
}
```

## ✅ Best Practices

- Always check errors explicitly — avoid silently discarding them with `_ = err`.
- Wrap errors with meaningful context using `%w`.
- Use `panic` only for genuinely unrecoverable cases — return an `error` for normal flow.
- Create custom error types only when the caller needs extra data from the error.

# 🟫 Day 12: Go Generics (Go 1.18+)

Generics let you write one function or struct that works across multiple types, without duplicating code.

## ⚙️ Generic Function

```
func Max[T int | float64](a, b T) T {
    if a > b {
        return a
    }
    return b
}

fmt.Println(Max(10, 20))   // works with int
fmt.Println(Max(3.5, 2.1)) // works with float64
```

## 📐 Constraints

A constraint defines which types a type parameter is allowed to accept.

```
type Number interface {
    int | int64 | float64
}

func Sum[T Number](nums []T) T {
    var total T
    for _, n := range nums {
        total += n
    }
    return total
}
```

The `constraints` package (`golang.org/x/exp/constraints`) already provides ready-made constraints like `Ordered`, `Signed`, and `Float`.

## 🧱 Generic Structs

```
type Stack[T any] struct {
    items []T
}

func (s *Stack[T]) Push(item T) {
    s.items = append(s.items, item)
}

func (s *Stack[T]) Pop() (T, bool) {
    var zero T
    if len(s.items) == 0 {
        return zero, false
    }
    last := s.items[len(s.items)-1]
    s.items = s.items[:len(s.items)-1]
    return last, true
}
```

```
intStack := Stack[int]{}
intStack.Push(10)
intStack.Push(20)
val, _ := intStack.Pop()
fmt.Println(val) // 20
```

## 🧠 When Should You Use Generics?

- When you've written the same logic 2–3 times for different types (a duplication signal).
- When building general-purpose data structures (stack, queue, linked list).
- When you need type safety without losing flexibility.

⚠️ Don't overuse them — for simple cases, a plain function or interface is usually more readable.

# 🟦 Day 13: Concurrency — Go's Superpower

This is the feature Go is famous for. Concurrency means managing multiple tasks at once — it doesn't necessarily mean running them in parallel.

> "Concurrency is about dealing with lots of things at once. Parallelism is about doing lots of things at once." — Rob Pike

## 🧵 Goroutines — Lightweight Threads

```
func processOrder(id int) {
    fmt.Println("Processing order:", id)
}

func main() {
    go processOrder(101) // starts a goroutine
    time.Sleep(time.Second) // use WaitGroup in production, not Sleep
}
```

A goroutine isn't an OS thread — it's managed by Go's own lightweight runtime scheduler, so you can run thousands of them without slowing down the system.

## 📡 Channels — Communication Between Goroutines

```
func worker(ch chan string) {
    ch <- "task completed"
}

func main() {
    ch := make(chan string)
    go worker(ch)

    result := <-ch
    fmt.Println(result)
}
```

### Buffered Channels

```
queue := make(chan int, 3) // can hold 3 items without blocking
queue <- 1
queue <- 2
queue <- 3
```

### Directional Channels

```
func sendOnly(ch chan<- string, msg string) {
    ch <- msg
}

func receiveOnly(ch <-chan string) string {
    return <-ch
}
```

## 🎛️ Select Statement

`select` monitors multiple channels at once and runs whichever one becomes ready first.

```
select {
case msg1 := <-ch1:
    fmt.Println("Received from ch1:", msg1)
case msg2 := <-ch2:
    fmt.Println("Received from ch2:", msg2)
default:
    fmt.Println("Nothing is ready yet")
}
```

## ⏳ WaitGroup — Tracking Goroutine Completion

```
var wg sync.WaitGroup

for i := 1; i <= 3; i++ {
    wg.Add(1)
    go func(id int) {
        defer wg.Done()
        fmt.Println("Task completed:", id)
    }(i)
}

wg.Wait() // blocks until all goroutines finish
```

## 🔒 Mutex & RWMutex — Keeping Shared Data Safe

```
type Counter struct {
    mu    sync.Mutex
    count int
}

func (c *Counter) Increment() {
    c.mu.Lock()
    defer c.mu.Unlock()
    c.count++
}
```

When multiple goroutines read/write the same variable simultaneously, without a `Mutex` you get a **race condition** — the result becomes unpredictable.

`RWMutex` is better when reads far outnumber writes — readers don't block each other.

## ⚠️ Deadlocks & Race Conditions

- **Deadlock:** Two goroutines are waiting on each other, and neither can proceed.
- **Race condition:** The result depends on timing — sometimes correct, sometimes not.

Go includes a built-in tool to detect race conditions:

```
go run -race main.go
```

# 🟩 Day 14: Advanced Concurrency Patterns

These patterns are usually combined together in real-world Go projects.

## 🌊 Fan-Out / Fan-In

**Fan-Out:** Splitting one task across multiple goroutines.
**Fan-In:** Merging results from multiple channels into a single channel.

```
func fanOut(jobs <-chan int, id int, results chan<- int) {
    for job := range jobs {
        results <- job * job
    }
}
```

## 👷 Worker Pool

A fixed number of workers pulling jobs from a queue — the best pattern for controlling resource usage.

```
func worker(id int, jobs <-chan int, results chan<- int) {
    for j := range jobs {
        results <- j * 2
    }
}

func main() {
    jobs := make(chan int, 100)
    results := make(chan int, 100)

    for w := 1; w <= 3; w++ {
        go worker(w, jobs, results)
    }

    for j := 1; j <= 5; j++ {
        jobs <- j
    }
    close(jobs)

    for a := 1; a <= 5; a++ {
        fmt.Println(<-results)
    }
}
```

This pattern lets you run scraping jobs, batch processing, or bulk API calls in a controlled way, without overloading the server.

## 🔗 Pipeline Pattern

Chained stages where the output of one stage becomes the input of the next:

```
func generate(nums ...int) <-chan int {
    out := make(chan int)
    go func() {
        defer close(out)
        for _, n := range nums {
            out <- n
        }
    }()
    return out
}

func square(in <-chan int) <-chan int {
    out := make(chan int)
    go func() {
        defer close(out)
        for n := range in {
            out <- n * n
        }
    }()
    return out
}
```

```
// usage
for result := range square(generate(1, 2, 3, 4)) {
    fmt.Println(result)
}
```

## ⏲️ Context — Cancellation and Timeouts

```
func fetchData(ctx context.Context) error {
    select {
    case <-time.After(3 * time.Second):
        fmt.Println("Data received")
        return nil
    case <-ctx.Done():
        return ctx.Err()
    }
}

func main() {
    ctx, cancel := context.WithTimeout(context.Background(), 2*time.Second)
    defer cancel()

    if err := fetchData(ctx); err != nil {
        fmt.Println("Request cancelled/timed out:", err)
    }
}
```

`context` is the standard way to handle cancellation and timeouts for API calls, database queries, and background jobs — you'll see it everywhere in production Go code.

## 🚦 Rate Limiting

```
limiter := time.Tick(200 * time.Millisecond)

for req := range requests {
    <-limiter // only one request is processed every 200ms
    process(req)
}
```

## ⚛️ Sync Package Recap

| Tool | Purpose |
|------|---------|
| `WaitGroup` | Track completion of goroutines |
| `Mutex` | Allow only one goroutine access at a time |
| `RWMutex` | Multiple readers, single writer |
| `Once` | Guarantee a function runs exactly once |
| `Pool` | Reuse temporary objects to reduce GC pressure |
| `atomic` | Lock-free, safe operations on small values |
